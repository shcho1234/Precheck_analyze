# PreCheck 로그포맷정의서 v1.0

# 문서의 목적
- 로그 포맷 정의서는 각 서브 시스템에서 생성하는 비정규화된 로그 중 정규화할 필요성이 있는(관리자가 의미를 부여하는) 로그를 입력 타입별로 나누고
각 로그의 표시 형식을 정의함
- 로그 입력 타입에 따라 분석 시스템에서 분석방법이 달라짐
- 관리의 편의성을 위해 로그 입력타입은 최소한으로 정의

---

# 공통 사항
- 정규화된 로그의 맨처음은 @@@[timestamp임, timestamp는 yyyy/MM/dd HH:mm:ss.SSS형식, 끝은 @@@로 끝남
- 정규화된 로그는 timestamp 다음에 [입력타입] 그 다음에 [로그식별코드(LOG_ID)]가 옴
- 예) @@@[2026/04/21 12:12:32.123][수치][FILE_COUNT]|파일갯수 3|$3$@@@
- 수치형 로그에서 값이 없는 경우 로그 생성 금지

## 로그 식별 코드 (LOG_ID)
- 각 정규화 로그는 고유한 식별 코드를 가짐
- LOG_ID는 분석 서버가 어떤 분석 정책을 적용할지 결정하는 핵심 키
- 형식 규칙
  -> 영문 대문자 + 숫자 + 언더스코어(_)만 사용
  -> 최대 30자 이내
  -> 공백, 한글, 특수문자 사용 금지
- 명명 규칙 권장 형식: [분류약어]_[항목약어]
  -> 예) DISK_HOME       : home 디스크 사용률
  -> 예) DISK_DATA       : data 디스크 사용률
  -> 예) PROC_COUNT      : 프로세스 수
  -> 예) PUSH_STATUS     : PUSH 전송 상태
  -> 예) MEM_MINI        : 미니원장 메모리 확인
  -> 예) FILE_CHECK      : 체크 파일 존재 여부
  -> 예) DATE_BDAY       : 영업일 날짜 확인
- LOG_ID는 각 서브 시스템 담당자가 정의하며, 분석 정책 정의서와 1:1 매칭됨
- 동일 LOG_ID가 여러 서버에서 생성될 수 있으나, 분석 정책은 서버 + LOG_ID 조합으로 구분
- 공백은 구분자로 사용하지 않는다, 필드 내부 공백은 그대로 유지한다
- 각 로그는 개행(\n)을 기준으로 구분한다
- 수치형 로그의 값은 정수 또는 실수로 표현한다
 - 정의된 포맷에 맞지 않는 로그는 수집 및 분석 대상에서 제외되므로 포맷정의서대로 로그가 생성되어야함
 - 정규화된 로그가 일반 로그 중간에 올수 있으므로 정규화로그의 시작과 끝은 @@@로 구분함
 - 정규화 로그 시작(@@@) 이후 종료(@@@)가 없는 경우 해당 로그는 무시한다.
 - 한라인에 중복된 정규화 로그가 있을수 없음
   예) 2024/06/01 sis15007 @@@[2026/04/21 12:12:32.123] [수치] [MINI_FAIL] |미니원장잔고처리실패수|$10$@@@
   예) @@@[2026/04/21 12:12:32.123] [수치] [MINI_FAIL] |미니원장잔고처리실패수|$10$@@@ 처리내역을 확인하세요.

# 로그 입력 타입
  - 로그 입력 타입 설명에는 @@@로 시작 @@@로 끝나는 부분 생략해서 설명함
  ## 문구형식
  - 의미 : 생성되는 자연어 문장을 기반으로한 로그
  - 표시방법 : "[timestamp]" + "[문구]" + "[LOG_ID]" + "|로그내용|"
  - 분석 방법 : LOG_ID에 매칭된 분석 정책의 에러 키워드가 로그 내용에 포함되면 에러로 판별, 포함되지 않으면 정상
  - 예) @@@[2026/04/21 12:12:32.123][문구][PUSH_STATUS]|PUSH 전송 실패(PUSH 서버 실패 O)|@@@
  -> LOG_ID=PUSH_STATUS의 에러 키워드 목록에 "실패"가 등록되어 있고 로그 내용에 "실패"가 포함되어 에러

  ## 정보형식
  - 의미 : 분석 없이 DB에 저장하여 Dashboard 표시용으로 사용하는 로그 
  - 표시방법 : "[timestamp]" + "[정보]" + "[LOG_ID]" + "|로그내용|"
  - 분석 방법 : 분석 없이 정보만 저장
  - 예) @@@[2026/04/21 12:12:32.123][정보][OCBTCH_RECV]|[OCBTCH]- 상품옵션기본 DATA 수신완료|@@@
  - 예) @@@[2026/04/21 12:12:32.123][정보][STOPLOSS_CNT]|스탑로스 계좌 수 : 482|@@@

  ## 날짜형식
  - 의미 : 날짜를 나타내기 위한 로그
  - 표시방법 : "[timestamp]" + "[날짜]" + "[LOG_ID]" + "|로그내용,로그내용중 yyyy/MM/dd형식 날짜 포함|", 로그내용 내 날짜는 yyyy/MM/dd 형식을 따른다
  - 분석 방법 : 로그내용의 날짜와 오늘 날짜와 같으면 정상, 다르면 에러, 로그내용을 모두 분석함
  - 예) @@@[2026/04/21 12:12:32.123][날짜][DATE_BDAY]|bday[2026/04/22] frec[12607] krec[3] fpos [07393]|@@@

  ## 수치형식
  - 의미 : 로그의 내용
  - 표시방법 : "[timestamp]" + "[수치]" + "[LOG_ID]" + "|로그내용|" + "$숫자$"
  - 분석 방법 : LOG_ID에 매칭된 분석 정책의 기준값과 >, >=, <, <=, = 연산 비교후 참이면 정상, 거짓이면 에러, 검사할 임계수치 근접에 도달했을경우 경고
  - 예) @@@[2026/04/21 12:12:32.123][수치][DISK_HOME]|home디스크|$80$@@@
  -> LOG_ID=DISK_HOME의 임계치가 90이고 연산자가 <인 경우, 80 < 90이므로 정상
  - 예) @@@[2026/04/21 12:12:32.123][수치][PUSH_KEY]|v_push키|$51769$@@@
  -> LOG_ID=PUSH_KEY의 임계치가 50000이고 연산자가 <인 경우, 51769 < 50000이 거짓이므로 에러
  - 레거시 시스템의 많은 수정이 필요함

  ## 존재형식
  - 의미: 파일, SHM등 감시 대상이 존재 하지 않으면 남는 로그
  - 표시방법 : "[timestamp]" + "[존재]" + "[LOG_ID]" + "|로그내용|"
  - 분석 방법 : 존재 형식은 감시 대상의 부존재때문에 남는 로그 이기 때문에 에러로 판별
  - 예) @@@[2026/04/21 12:12:32.123][존재][MEM_MBSOSI]|mbsosi메모리 없음|@@@
