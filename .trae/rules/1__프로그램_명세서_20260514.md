# 장전 서버 점검 자동화 시스템 개발 Context v1.0

## [프로젝트 목적]
- 레거시 C 기반 시스템/서버들의 운영 로그를 표준화하여 수집·분석하고, 이상 징후를 관리자에게 통보하는 통합 모니터링 시스템
- 개발 사항은 [로그 수집 서버] [로그 분석 서버] [결과 통보 서버] [DashBoard]

---

## [공통]
- 로그 수집 지연 허용시간 : 실패시 10초 간격 3회 시도(최초 1회 + 실패후 재시도 3회)
- 최대 대상 시스템 수 : 100
- 일일 로그 건수 : 500건 미만
- 보관 기간 : 수집된 파일 10일 저장, DB 데이터는 별도 보관 정책을 따름(별도로 삭제할예정)
- 장애 시 복구 기준 : 정식 DR/HA 구조는 1차 버전 범위에서 제외, 단건 실패 재시도 및 실패 로그 기록은 포함한다
- 외부 접속 보안 수준 : 외부 접속 없으므로 고려 하지 않음(향후 접속 가능하게 수정할 예정, 1차 버전은 내부망 전용으로 가정)
- 시스템 자체 로그 : log4j 사용하여 각 시스템 특정 디렉토리에 시스템명.yyyymmddHHMMssSSS.log로 남김
- 스케쥴, 임계치, 임계근접치, 문구 등은 1차 버전은 파일 기반 설정으로 구현하고, 향후 관리자 기능 개발 시 DB 기반으로 전환한다
- 서버 시간은 동일한 시스템 시간대를 사용한다
- 로그 파일의 timestamp 표기는 yyyy/MM/dd HH:mm:ss.SSS 형식을 사용한다, DB 내부 시간 컬럼은 표준 datetime/timestamp 타입 사용을 고려한다
- 수집, 분석 및 통보 실패는 시스템 로그에 기록하고 수동 재처리를 전제로 한다

## [DB공통]
- DB는 운영은 Altibase 사용, 테스트는 PostgreSQL 사용
- SQL과 DB 스키마는 Altibase와 PostgreSQL 공통으로 사용하게 작성
- Altibase는 원격 시스템에 있음
- 테스트를 위한 PostgreSQL은 로컬에 있음

## [공통 시스템간 경계]
- 로그 수집 서버는 정규화된 로그를 수신/저장만 하며 분석은 수행하지 않는다
- 로그 분석 서버는 저장된 로그를 판정만 하며 SMS 발송은 수행하지 않는다
- 결과 통보 서버는 통보 요청만 생성하며 통보 정책 판단은 레거시 시스템에 위임한다
- Dashboard는 조회 전용이며 운영 설정 변경은 수행하지 않는다
- 수집 서버도 스케줄 있음
- 분석 서버도 스케줄 있음

### (로그 정규화)
- 각 서브 시스템별 로그 정규화과정이 필요함, 지금은 정규화 되어 있지 않지만 각 서비스별 담당자가 정규화 시켜야 함
- 수집할 정보는 특정 입력 타입으로 저장
- 정규화 로그는 1줄 1건을 원칙으로 한다
- 정규화 로그는 [입력타입] 다음에 [로그식별코드(LOG_ID)]를 포함함, LOG_ID는 분석 정책 매칭의 핵심 키
- 정규화된 로그 포맷은 일반 로그의 중간에 올수 있음, 정규화 로그의 시작과 끝을 @@@로 구분함
  예) 2024/06/01 sis15007 @@@[2026/04/21 12:12:32.123][수치][MINI_FAIL]|미니원장잔고처리실패수|$10$@@@

## [로그 수집 서버]
- 각 서브시스템에서 정해진 시간 또는 주기별로 로그 수집, 정해진 시간에 수집하고 주기별로 또다시 수집할경우 시스템은 기존 수집된 정보 이후에 쌓은 로그만을 신규 로그로 인식함
- 신규로그는 파일의 마지막으로 읽은 라인번호로 판별함
- 각 서브시스템별로 최초 수집 시간 및 주기는 파일 형태의 스케쥴표로 관리함(로그수집스케쥴정의서)
- 로그 수집시도 실패시 10초간격으로 다시 시도
- 최초 시도 실패 후 재시도 3회 수행함(최초 1회 + 실패 후 재시도 3회), 재시도 3회 모두 실패시 시스템 에러 로그 남김
- 로그 수집은 로그 파일전체를 수집 후 수집대상 로그(@@@로시작하는 정규화 로그)만 추출하여 저장
- 최초 수집 사이즈가 300MB이상일 경우 수집하지 않고 수집 불가 로그를 남김, 해당 대상은 수집대상에서 영구 제외(운영자 수동 조치 필요)
- 부분 수집 사이즈가 50MB이상일 경우 수집하지 않고 수집 불가 로그를 남김, 해당 대상은 수집대상에서 영구 제외(운영자 수동 조치 필요)
- 각 서브시스템에서 수집한 로그는 서버구분/시간별/입력 타입별로 DB에 저장
- 주기별 로그 수집은 주기별로 로그 수집 후 DB insert

## [로그 분석 서버]
- 로그 분석 서버는 정해진 시간(스케쥴, 최초 및 주기별 스케쥴)에 DB에 저장된 각 서브시스템별 로그를 분석함
- 분석은 각 시스템별/로그별 정량적 분석을 하며, 임계수치 또는 문구는 1차 버전에서는 파일 기반으로 관리한다  
  (분석은 로그의 제한된 정규화 방식을 처리하며 사전 정의된 문자열 규칙 및 수치 비교 규칙에 따라 판정하여 분석함)
- 스케쥴, 임계수치 또는 임계수치 근접 비율, 문구등은 파일 형태로 관리하며 분석 서버는 이파일을 읽는다
- 분석 결과는 분석 레벨 별로 DB에 저장, 결과 통보 서버에서 활용함
- 분석 레벨은 [정상][에러][경고][정보][미분석]으로 구분
- 분석 시간 및 주기는 파일 형태의 스케쥴표로 관리함(로그분석스케쥴정의서)

### (분석 레벨)
- [정상] -> 분석 정책에 합당한 경우
- [에러] -> [존재]는 에러, [수치]는 조건식에 맞지 않을경우
- [경고] -> 임계치를 넘지 않았지만 임계치 근접한경우, 수치형에만 해당
- [정보] -> 분석 레벨이 아닌 저장용 타입이며 분석 결과 레벨에 포함되지 않는다
- [미분석] -> 분석 정책 파일에 LOG_ID가 등록되지 않은 경우, 운영자가 정책 파일 업데이트 필요, 통보 대상 제외

#### 입력 타입별 분석 가능 레벨

| 입력 타입 | 정상 | 경고 | 에러 |
|---|---|---|---|
| 문구 | ✅ | ❌ | ✅ |
| 수치 | ✅ | ✅ | ✅ |
| 존재 | ❌ | ❌ | ✅ |
| 날짜 | ✅ | ❌ | ✅ |
| 정보 | - | - | - |

### (분석 정책)
- 모든 분석은 로그의 LOG_ID(로그 식별 코드)를 기준으로 분석 정책 파일에서 매칭함
- 동일 LOG_ID라도 서버별로 임계치가 다를 수 있어, 분석 정책은 [서버구분 + LOG_ID] 조합으로 관리함
- 분석 정책 파일에 등록되지 않은 LOG_ID의 로그는 ANALYZE_LEVEL='미분석'으로 DB에 저장 (시스템 로그에도 WARN 기록)
- [문구] : LOG_ID에 매칭된 에러 키워드를 포함하면 에러
  (추후 개발에서 사전 정의된 문자열 패턴과 매핑된 분석 레벨에 따라 [정상]/[에러]/[경고]를 판정한다로 수정예정)
- [수치] : LOG_ID에 매칭된 임계수치와 >, >=, <, <=, = 연산 비교후 참이면 정상, 거짓이면 에러, 검사할 임계수치 근접에 도달했을경우 경고
- [존재] : 존재는 부존재일경우에만 로그가 쌓이므로 에러
- [정보] : 정보성이므로 DB에 저장만, 정상, 에러 판단 안함, DashBoard에서 활용 예정
- [날짜] : 로그내용중 yyyy/MM/dd형식의 날짜가 오늘과 같다면 정상, 아니면 에러
- 분석 우선 순위는 입력 타입별로 정해져 있어 우선 순위는 존재 하지 않음, 단일 정책 비교만(동일 로그가 여러 규칙에 걸릴경우 없음)
- 분석 실행 단위는 1건단위임 시간/건수별 합산 없음

### (분석 결과 예)
- [에러] [PROC_COUNT] 프로세스수 10 < 20(임계치)
- [에러] [FILE_CHECK] test.txt 파일이 존재하지 않음
- [경고] [STOCK_COUNT] 종목수 90 < 100(임계치 대비 20% 근접)

## [결과 통보 서버 - 2차 개발에서 진행하므로 개발에서 제외]
- 정해진 시간에 DB에 저장되어 있는 분석결과를 관리자에게 통보함
- 통보하는 정보는 [에러], [경고]로 한정 나머지는 DashBoard에서 확인 하게함
- 통보 방법은 sms로 처리, sms 발송은 레거시 시스템을 통함(DB Table insert 방식)
- 통보 정책 즉 통보 중복 방지 수신자그룹, 시간, 휴일 유무, 재전송 여부는 레거시 시스템에서 처리함
- 통보 서버는 DB Insert만 처리

### (기술스택)
| 구분 | 권장 기술 | 선정 이유 |
|---|---|---|
| 개발 언어 | Java 17 | 전체 서버 기술 통일 |
| Framework | Spring Boot 3.x | 통보 요청 생성 서버로 구현하기 단순함 |
| 스케줄 처리 | Spring Scheduler (fixedDelay) | 이전 실행 완료 후 다음 실행 시작, 중복 통보 방지에 유리 |
| DB 접근 (PreCheck) | MyBatis (Mapper 방식) | 미통보 에러/경고 조회, NOTIFY_YN 업데이트 SQL 명확히 제어 가능 |
| DB 접근 (레거시 SMS) | Spring JdbcTemplate | 레거시 SMS 테이블은 단순 INSERT만 필요, JdbcTemplate으로 충분하고 MyBatis보다 설정이 가벼움 |
| 다중 DataSource | Spring Boot Multi DataSource | PreCheck DB와 레거시 SMS DB를 동시에 연결해야 하므로 DataSource 분리 설정 필요 |
| 외부 연계 | 기존 레거시 SMS/PUSH 시스템 연계 | 통보 정책 판단은 기존 시스템에 위임 |

## [DashBoard  - 2차 개발에서 진행하므로 개발에서 제외]
- Web base의 DashBoard를 제공
- DashBoard는 각 시스템별, 시간별, 입력 타입별 점검 결과를 화면에 보여줌
- 화면 새로 고침을 하지 않는 경우에도 5분마다 한번씩 자동 새로 고침을 함(로그 분석은 하지 않음)
- DashBoard는 날짜별 점검 결과를 보여줌(점검 히스토리)
- DashBoard는 날짜별 디스크 사용량, 종목수, 특화컨텐츠 사용자수 등을 그래프로 보여줌([수치] 정보를 활용 [정보]분석결과를 사용)
- DashBoard는 조회 전용이며 향후 관리자 기능까지 개발할 예정, 당장은 조회만
- 화면 디자인 및 기능, 목록은 추후 제공예정

### (기술스택)
| 구분 | 권장 기술 | 선정 이유 |
|---|---|---|
| Backend | Spring Boot 3.x | 기존 서버와 기술 통일 |
| 화면 | Thymeleaf + Bootstrap | React/Vue보다 단순하고 유지보수 난이도가 낮음, 별도 프론트 빌드 환경 불필요 |
| DB 접근 | MyBatis (Mapper 방식) | 날짜/서버/타입별 조회 SQL 작성이 명확함, 다른 서버와 기술 통일 |
| 차트 | Chart.js (CDN) | 가볍고 사용법이 단순함, CDN으로 설치 없이 사용 가능 |
| 자동 갱신 | JavaScript setInterval | 5분 자동 새로 고침, 별도 라이브러리 불필요 |
| 권한 | 1차 버전 제외 또는 단순 사내망 접근 제어 | 내부망 전용 기준으로 복잡도 최소화 |
