# PreCheck 로그 분석 스케쥴정의서 v1.0

## 문서의 목적
- 로그 분석 시스템에서 참고할 분석 대상서버, 대상파일 및 분석 주기를 정의한 파일을 설명
- 로그 분석 시스템내 파일 형태로 정의함

---

## 로그 분석 정의서 파일명
- 로그 분석 시스템 내 ~/cfg/PreCheck_AnalyzeLogs_Schedule.conf

---

## 분석 대상 지정 방식

분석 스케줄은 아래 두 가지 방식으로 대상을 지정할 수 있다.

| 방식 | 설명 | 형식 |
|---|---|---|
| 서버 단위 | 특정 서버에서 수집된 모든 로그를 분석 대상으로 지정 | [서버구분] 만 사용 |
| 파일 단위 | 특정 서버의 특정 파일에서 수집된 로그만 분석 대상으로 지정 | [서버구분][대상파일명] 함께 사용 |

> ⚠️ **중요 규칙**
> - 대상파일은 반드시 대상서버가 함께 지정되어야 함
> - 대상서버는 IP주소가 아닌 **서버구분(서버사용명)** 으로 식별함
>   (수집 스케줄 정의서의 [서버구분] 필드값과 동일한 값 사용)
> - 동일한 [서버구분 + 대상파일명] 조합이 중복되면 마지막 라인이 적용됨

## 로그 분석 주기

| 구분 | 설명 |
|---|---|
| 배치 | 일회성 분석. 수집된 대상 로그 전체를 한번만 분석함 |
| 주기 | 정해진 시간부터 주기별로 분석. 최초 분석은 대상 로그 전체, 이후 주기부터는 마지막으로 분석한 로그 다음부터 분석 (중복 분석 방지) |

### 배치 분석 동작 방식
```
[배치 스케줄 실행 시각]
        |
        ↓
TB_COLLECT_LOG에서 해당 서버(+파일) 의 수집 로그 전체 조회
        |
        ↓
TB_ANALYZE_RESULT에 이미 분석된 COLLECT_LOG_ID 제외
        |
        ↓
미분석 로그 전체 분석 수행
        |
        ↓
TB_ANALYZE_RESULT INSERT
TB_ANALYZE_HISTORY INSERT (배치 실행 이력)
```

### 주기 분석 동작 방식
```
[주기 스케줄 실행 시각]
        |
        ↓
TB_ANALYZE_HISTORY에서 해당 서버(+파일)의 마지막 분석 성공 이력 조회
→ LAST_ANALYZE_LOG_ID 확인 (마지막으로 분석한 COLLECT_LOG_ID)
        |
        ↓
TB_COLLECT_LOG에서 LAST_ANALYZE_LOG_ID 이후의 수집 로그만 조회
(COLLECT_LOG_ID > LAST_ANALYZE_LOG_ID 조건)
→ 최초 실행 시 LAST_ANALYZE_LOG_ID가 없으면 전체 조회
        |
        ↓
신규 수집 로그 분석 수행 (중복 분석 방지)
        |
        ↓
TB_ANALYZE_RESULT INSERT
TB_ANALYZE_HISTORY INSERT (주기 실행 이력 + LAST_ANALYZE_LOG_ID 갱신)
```

> 💡 **중복 분석 방지 원칙**
> - 주기 분석은 TB_ANALYZE_HISTORY의 LAST_ANALYZE_LOG_ID를 기준으로
>   마지막 분석 이후 수집된 로그만 분석함
> - 분석 서버가 재시작되어도 LAST_ANALYZE_LOG_ID를 DB에서 복구하여
>   안전하게 이어서 분석 가능

## 로그 분석 정의 파일 포맷

분석 정의 파일의 포맷은 대상 지정 방식에 따라 두 가지로 구분된다.

### 포맷 1: 서버 단위 분석
```
[서버구분][분석주기기술]
```

### 포맷 2: 파일 단위 분석
```
[서버구분][대상파일명(절대경로)][분석주기기술]
```

**필드 설명:**

| 필드 | 설명 |
|---|---|
| 서버구분 | 분석 대상 서버의 구분명. 수집 스케줄 정의서의 [서버구분]과 동일한 값 사용 |
| 대상파일명 | 수집 스케줄 정의서의 [대상파일명]과 동일한 값 사용. 생략 시 서버 전체 대상 |
| 분석주기기술 | 분석 구분, 분석 시간, 분석 주기 |

---

## 분석주기기술 포맷

수집 스케줄 정의서와 동일한 요일/시간 표기 방식을 사용한다.

```
배치 : [배치|요일|시작시간]
주기 : [주기|요일|시작시간|분석간격(분)|종료시간]
```

**요일 표기 규칙 (수집 스케줄 정의서와 동일)**

| 표기 | 의미 |
|---|---|
| 0~6 단일값 | 0->일요일, 1->월요일, ..., 6->토요일 |
| 시작-끝 범위 | 예) 1-5 = 월~금 (시작 ≤ 끝 조건) |
| * | 매일 (0-6과 동일) |
| 콤마(,) 또는 복합 표기 | 1차 버전 미지원, 해당 스케줄 무시 처리 |

**시간 표기 규칙**

| 필드 | 형식 | 범위 |
|---|---|---|
| 시작시간 | HHmmss | 000001 ~ 235959 |
| 분석간격(분) | 숫자 (분 단위) | 예) 10 = 10분 간격 |
| 종료시간 | HHmmss | 000001 ~ 235959 |

---

## 파일 공통 규칙

- 파일은 텍스트 형태이며 개행문자(\n)로 구분
- 위 포맷을 따르지 않는 스케줄은 무시함
- skip 처리는 #으로 시작함
- 대상파일명 없이 서버구분만 지정 시 해당 서버의 모든 수집 로그를 분석 대상으로 함
- 동일 [서버구분 + 대상파일명] 조합이 중복되면 마지막 라인이 적용됨
- 분석 실패 시 재시도 정책은 프로그램 명세서를 따름 (최초 시도 실패 후 5분 간격 재시도 3회, 배치/주기 공통 적용)

## 스케쥴 예

### 서버 단위 분석 예시
```
# 서버 전체 대상 - 배치
[dlprem01-테스트개발][배치|1-5|093001]
→ 월~금요일 9시 30분 01초에 dlprem01-테스트개발 서버의 수집 로그 전체를 한번 분석

[orderserver-자동주문][배치|*|100001]
→ 매일 10시 00분 01초에 orderserver-자동주문 서버의 수집 로그 전체를 한번 분석

# 서버 전체 대상 - 주기
[dlprem01-테스트개발][주기|1-5|090001|30|180001]
→ 월~금요일 9시 00분 01초부터 30분 간격으로 18시 00분 01초까지
  dlprem01-테스트개발 서버의 신규 수집 로그를 주기적으로 분석
```

### 파일 단위 분석 예시
```
# 특정 파일 대상 - 배치
[dlprem01-테스트개발][/tmp/check.out][배치|1-5|093001]
→ 월~금요일 9시 30분 01초에 /tmp/check.out 파일에서 수집된 로그 전체를 한번 분석

# 특정 파일 대상 - 주기
[dlprem01-테스트개발][/tmp/check.out][주기|*|090001|10|100001]
→ 매일 9시 00분 01초부터 10분 간격으로 10시 00분 01초까지
  /tmp/check.out 파일에서 수집된 신규 로그를 주기적으로 분석

[dlprem01-테스트개발][/var/log/app/service.log][주기|1-5|090001|10|180001]
→ 월~금요일 9시 00분 01초부터 10분 간격으로 18시 00분 01초까지
  /var/log/app/service.log 파일에서 수집된 신규 로그를 주기적으로 분석
```

### skip 처리 예시
```
#[dlprem01-테스트개발][배치|*|090001] -> skip 처리됨
```

## 수집 스케줄과의 관계

분석 스케줄은 수집 스케줄과 독립적으로 운영되며, 아래 관계를 참고하여 설정한다.

```
[수집 스케줄]                     [분석 스케줄]
dlprem01-테스트개발               dlprem01-테스트개발
/tmp/check.out                    /tmp/check.out
주기 | * | 09:00 | 10분 | 18:00   주기 | * | 09:05 | 10분 | 18:05
        |                                 |
        |     수집 완료 후 5분 뒤          |
        └──────────────────────────────>──┘
                                  수집된 로그를 분석
```

> 💡 **수집 → 분석 시간 간격 권장사항**
> - 분석 스케줄의 시작시간은 수집 스케줄의 시작시간보다 **5분 이상 늦게** 설정 권장
> - 수집이 완료되기 전에 분석이 실행되면 해당 주기의 로그를 분석하지 못할 수 있음
> - 이 경우 다음 주기에 해당 로그가 자동으로 분석됨 (중복 분석 방지 로직으로 보장)

## 수집 스케줄 정의서와의 차이점 요약

| 항목 | 수집 스케줄 정의서 | 분석 스케줄 정의서 |
|---|---|---|
| 대상 식별 | 서버구분 + IP주소 + 파일경로 | 서버구분 + 파일경로 (IP 없음) |
| 대상 지정 방식 | 파일 단위만 가능 | 서버 단위 또는 파일 단위 |
| 주기 기준 | 마지막으로 읽은 파일 라인번호 | 마지막으로 분석한 COLLECT_LOG_ID |
| 중복 방지 수단 | TB_COLLECT_HISTORY.LAST_LINE_NUMBER | TB_ANALYZE_HISTORY.LAST_ANALYZE_LOG_ID |
| 재시도 정책 | 명세서 기준 동일 | 명세서 기준 동일 |
