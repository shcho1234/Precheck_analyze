# PreCheck 로그 분석 정책 정의서 v1.0

---

## 1. 문서 목적

이 문서는 PreCheck 로그 분석 서버에서 사용하는 **분석 정책 파일**의 포맷을 정의한다.  
분석 서버는 수집된 정규화 로그의 **LOG_ID(로그 식별 코드)**를 기준으로 이 파일을 조회하여  
임계치 비교, 에러 키워드 매칭 등 분석 작업을 수행한다.

> 💡 1차 버전은 파일 기반으로 관리하며, 향후 관리자 기능 개발 시 DB 기반으로 전환 예정

---

## 2. 분석 정책 파일 위치

- 분석서버 루트디렉토리에서 ~/cfg/PreCheck_AnalyzePolicy.conf

---

## 3. 분석 정책 파일 공통 규칙

- 파일은 텍스트 형태이며 개행문자(\n)로 구분
- 위 포맷을 따르지 않는 정책은 무시함
- 주석 처리는 라인 시작에 #으로 표시
- 동일 [서버구분 + LOG_ID] 조합이 중복되면 마지막 라인이 적용됨
- 정책 파일에 등록되지 않은 LOG_ID의 로그 처리 (방식 B 적용):
  -> TB_ANALYZE_RESULT에 ANALYZE_LEVEL='미분석'으로 INSERT
  -> ANALYZE_MESSAGE에 "[미분석][{LOG_ID}] 분석 정책 미등록" 형식으로 기록
  -> 시스템 로그(log4j)에 WARN 레벨로 기록
  -> 운영자는 미분석 로그를 확인하여 정책 파일 업데이트
  -> 미분석 로그는 결과 통보 서버의 통보 대상에서 제외 (에러/경고만 통보)
- 파일은 분석 서버 시작 시 1회 로딩, 운영 중 변경 시 분석 서버 재기동 필요  
  (향후 동적 리로드 기능 추가 예정)

## 4. 입력 타입별 정책 포맷

분석 정책은 입력 타입별로 포맷이 다르다.  
모든 정책의 공통 시작 부분은 `[서버구분][LOG_ID][입력타입]`이다.

### 4-1. 문구형 정책 포맷

```
[서버구분][LOG_ID][문구][에러키워드1,에러키워드2,...]
```

**필드 설명:**

| 필드 | 설명 | 예시 |
|---|---|---|
| 서버구분 | 대상 서버 구분명 (스케줄 정의서와 동일) | `dlprem01-테스트개발` |
| LOG_ID | 로그 식별 코드 | `PUSH_STATUS` |
| 입력타입 | 항상 `문구` | `문구` |
| 에러키워드 | 콤마로 구분된 키워드 목록, 로그 내용에 하나라도 포함되면 에러 | `실패,오류,Fail` |

**예시:**

```
[dlprem01-테스트개발][PUSH_STATUS][문구][실패,오류,Fail]
[dlprem01-테스트개발][LOGIN_STATUS][문구][denied,refused,timeout]
```

---

### 4-2. 수치형 정책 포맷

```
[서버구분][LOG_ID][수치][연산자][임계치][경고근접비율]
```

**필드 설명:**

| 필드 | 설명 | 예시 |
|---|---|---|
| 서버구분 | 대상 서버 구분명 | `dlprem01-테스트개발` |
| LOG_ID | 로그 식별 코드 | `DISK_HOME` |
| 입력타입 | 항상 `수치` | `수치` |
| 연산자 | 비교 연산자 | `>`, `>=`, `<`, `<=`, `=` |
| 임계치 | 비교 기준값 (정수 또는 실수) | `90` |
| 경고근접비율 | 임계치 대비 경고 판정 비율 (%) | `20` |

**판정 로직:**
- 로그값 `OPERATOR` 임계치 → 참이면 **정상**
- 거짓이지만 임계치 대비 경고비율 이내 근접 → **경고**
- 거짓이고 경고비율도 초과 → **에러**

**예시:**

```
[dlprem01-테스트개발][DISK_HOME][수치][<][90][20]
  -> home디스크 사용률이 90 미만이면 정상
  -> 90 이상이지만 임계치 90의 20%(=72) 이내인 경우 경고
  -> 단, 연산자가 < 이면 "값이 임계치 미만이어야 정상"이므로
     경고는 "값 >= 임계치 - 임계치*20% = 72" 인 영역에서 발생

[dlprem01-테스트개발][PROC_COUNT][수치][>][20][10]
  -> 프로세스 수가 20 초과면 정상
  -> 20 이하인데 임계치 20의 10%(=2) 이내(즉 18~20)인 경우 경고
  -> 18 미만이면 에러
```

> 💡 **경고 근접 판정 상세 로직은 분석 서버 구현 시 명확히 정의 필요**  
> 연산자 방향(< / >)에 따라 경고 영역 계산이 다르므로  
> 분석 서버 코딩 시 단위 테스트로 검증할 것

### 4-3. 날짜형 정책 포맷

```
[서버구분][LOG_ID][날짜]
```

**필드 설명:**

| 필드 | 설명 | 예시 |
|---|---|---|
| 서버구분 | 대상 서버 구분명 | `dlprem01-테스트개발` |
| LOG_ID | 로그 식별 코드 | `DATE_BDAY` |
| 입력타입 | 항상 `날짜` | `날짜` |

**판정 로직:**
- 로그 내용에 포함된 yyyy/MM/dd 형식의 날짜가 오늘과 같으면 **정상**
- 다르면 **에러**
- 로그 내용 중 모든 날짜를 검사함

**예시:**

```
[dlprem01-테스트개발][DATE_BDAY][날짜]
  -> 추가 임계치 없음, LOG_ID 매칭만 확인 후 날짜 비교 수행
```

---

### 4-4. 존재형 정책 포맷

```
[서버구분][LOG_ID][존재]
```

**필드 설명:**

| 필드 | 설명 | 예시 |
|---|---|---|
| 서버구분 | 대상 서버 구분명 | `dlprem01-테스트개발` |
| LOG_ID | 로그 식별 코드 | `MEM_MBSOSI` |
| 입력타입 | 항상 `존재` | `존재` |

**판정 로직:**
- 로그가 존재한다는 것 자체가 감시 대상의 부존재를 의미하므로 **항상 에러**

**예시:**

```
[dlprem01-테스트개발][MEM_MBSOSI][존재]
  -> mbsosi메모리 부존재 로그가 발생하면 무조건 에러
```

---

### 4-5. 정보형 정책 포맷

```
[서버구분][LOG_ID][정보]
```

**필드 설명:**

| 필드 | 설명 | 예시 |
|---|---|---|
| 서버구분 | 대상 서버 구분명 | `dlprem01-테스트개발` |
| LOG_ID | 로그 식별 코드 | `STOPLOSS_CNT` |
| 입력타입 | 항상 `정보` | `정보` |

**판정 로직:**
- 분석하지 않고 DB에 저장만 함
- Dashboard에서 그래프 등으로 활용

**예시:**

```
[dlprem01-테스트개발][STOPLOSS_CNT][정보]
  -> 스탑로스 계좌 수 로그를 DB에 저장만 함
```

## 5. 분석 정책 파일 전체 예시

```
# ============================================================
# PreCheck 분석 정책 정의 파일
# 형식: [서버구분][LOG_ID][입력타입][...타입별 필드...]
# ============================================================

# --- 디스크 사용률 모니터링 (수치형) ---
[dlprem01-테스트개발][DISK_HOME][수치][<][90][20]
[dlprem01-테스트개발][DISK_DATA][수치][<][85][15]

# --- 프로세스 수 모니터링 (수치형) ---
[dlprem01-테스트개발][PROC_COUNT][수치][>][20][10]

# --- 메시지 큐 사이즈 모니터링 (수치형) ---
[dlprem01-테스트개발][PUSH_KEY][수치][<][50000][10]

# --- 문구 기반 에러 감시 (문구형) ---
[dlprem01-테스트개발][PUSH_STATUS][문구][실패,오류,Fail]
[dlprem01-테스트개발][LOGIN_STATUS][문구][denied,refused,timeout]

# --- 영업일 날짜 확인 (날짜형) ---
[dlprem01-테스트개발][DATE_BDAY][날짜]

# --- 메모리/파일 존재 감시 (존재형) ---
[dlprem01-테스트개발][MEM_MBSOSI][존재]
[dlprem01-테스트개발][FILE_CHECK][존재]

# --- Dashboard용 정보 저장 (정보형) ---
[dlprem01-테스트개발][STOPLOSS_CNT][정보]
[dlprem01-테스트개발][OCBTCH_RECV][정보]

# --- 다른 서버 정책 ---
[orderserver-자동주문][DISK_HOME][수치][<][95][10]
[orderserver-자동주문][PROC_COUNT][수치][>][30][10]
```

## 6. 분석 정책 매칭 로직

분석 서버가 수집 로그 1건을 분석할 때의 처리 흐름:

```
[수집 로그 1건 읽음]
  ↓
[SERVER_ID + LOG_ID + LOG_TYPE]로 분석 정책 검색
  ↓
정책 있음? ────── 없음 → ANALYZE_LEVEL='미분석'으로 INSERT
  ↓ 있음            + 시스템 로그 WARN 기록
                    + 통보 서버 대상에서 제외
[입력 타입별 분석 수행]
  - 문구 → 키워드 매칭
  - 수치 → 임계치 비교 + 경고 판정
  - 날짜 → 오늘 날짜 비교
  - 존재 → 항상 에러
  - 정보 → 분석 없음
  ↓
[TB_ANALYZE_RESULT INSERT]
```

## 7. 분석 정책과 다른 문서의 관계

```
[로그 포맷 정의서]                  [분석 정책 정의서]
정규화 로그에 LOG_ID 포함  ──매칭──> 정책의 LOG_ID
        |
        ↓
[수집 DB]
TB_COLLECT_LOG.LOG_ID 저장
        |
        ↓
[분석 서버]
SERVER_ID + LOG_ID로 정책 검색
        |
        ↓
[분석 DB]
TB_ANALYZE_RESULT.LOG_ID 저장 (추적 가능)
```

## 8. LOG_ID 명명 규칙 (권장)

- 영문 대문자 + 숫자 + 언더스코어(_)만 사용
- 최대 30자 이내
- `[분류약어]_[항목약어]` 형식 권장

**분류 약어 예시:**

| 약어 | 의미 |
|---|---|
| DISK | 디스크 관련 |
| PROC | 프로세스 관련 |
| MEM | 메모리 관련 |
| FILE | 파일 관련 |
| PUSH | PUSH 관련 |
| DATE | 날짜 관련 |
| LOGIN | 로그인 관련 |
| MSG | 메시지 큐 관련 |

**전체 예시:**

| LOG_ID | 의미 |
|---|---|
| DISK_HOME | home 디스크 사용률 |
| DISK_DATA | data 디스크 사용률 |
| PROC_COUNT | 프로세스 수 |
| MEM_MINI | 미니원장 메모리 |
| FILE_CHECK | 체크 파일 존재 |
| PUSH_STATUS | PUSH 전송 상태 |
| PUSH_KEY | PUSH 키 카운트 |
| DATE_BDAY | 영업일 날짜 확인 |
| LOGIN_STATUS | 로그인 상태 |

## 9. 현재 설계 범위 및 향후 계획

| 항목 | 1차 버전 | 향후 |
|---|---|---|
| 정책 관리 방식 | 파일 (텍스트) | DB 테이블 (관리자 기능) |
| 정책 변경 적용 | 서버 재기동 필요 | 동적 리로드 |
| 다중 정책 매칭 | 1:1 매칭만 지원 | 우선순위 기반 다중 매칭 |
| 정책 이력 관리 | 미지원 | 정책 변경 이력 추적 |

## 10. 분석 정책 작성 시 주의사항

- **LOG_ID는 로그 포맷 정의서의 규칙을 반드시 준수해야 함**
- **수치형 정책의 연산자 방향에 따라 경고 영역 계산이 달라짐**, 구현 시 주의
- **동일 LOG_ID라도 서버별로 다른 임계치 적용 가능**
- 정책 파일이 잘못 작성되면 분석 서버가 시작 시 에러 발생  
  (시스템 로그에 어떤 라인이 잘못되었는지 기록됨)
- **운영 중 정책 변경 시 분석 서버 재기동 필요**, 변경 전 사전 검토 필수
